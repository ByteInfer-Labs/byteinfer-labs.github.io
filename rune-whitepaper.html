ByteInfer Rune: safe graph surgery for ONNX models_

A rule language for rewriting ONNX graphs. It finds every site a pattern matches, refuses any rewrite that would leave the graph broken, and ships with validation and profiling tools to check what a change did. This paper walks through the system and four jobs it has done on real models, starting with the strongest one, adding trainable LoRA adapters to an exported LLM.

1The problem

If you deploy ONNX models, sooner or later you have to edit one. The runtime fuses the operator patterns it recognizes and nothing else. When your model contains an idiom outside that set, say a LayerNorm exported from TensorFlow as thirteen primitive ops, or an attention block that should become a single fused kernel, the standard answer is a Python script that edits the protobuf directly.

Those scripts fail in a frustrating way. Graph edits are non-local. Delete a node and miss one of its consumers, and the model still serializes and still passes a schema check. The failure shows up at session build, as a runtime error naming a tensor nowhere near the line you got wrong. The script takes an afternoon to write. Chasing that tensor back to the actual mistake is where the week goes.

ByteInfer Rune exists to make this class of work boring. You write a pattern and a rewrite. The engine finds the sites, simulates the change, checks that the result is still a complete graph, and applies it only if the check passes. Anything it cannot prove safe, it refuses, and tells you which rule, which node, and which port.

2Why not just write a script?

For one edit at one site, a hand-written script is fine. The case for a rule engine builds as the work grows.

  • Coverage. A pattern finds every site, including the ones you did not know about. The LoRA rules hit 60 injection sites and the attention rewrite matched 120 times across 30 layers. Nobody audits that by eye.
  • Bad rewrites are caught before they land. The gate refuses at the exact site, names the rule, node, and port, and leaves the graph untouched, so a rewrite that would corrupt a residual connection is stopped at author time instead of surfacing at inference.
  • Rules are short enough to review. The LayerNorm fusion below is twelve lines and reads as a statement of intent. The equivalent protobuf script buries the same intent in bookkeeping, and reviewers stop reading.
  • Rules transfer between models. A fusion written against one layer applies to any model with the same export dialect.
  • Checking tools are built in. rune validate compares a rewritten model against the original for numerical equivalence, and rune profile shows where the time goes on CPU or GPU, down to a Chrome trace.

3What the DSL does

A rule is a pattern plus commands. The pattern names a chain of operators and gives each one an alias you can refer to later. Attribute constraints and alternation narrow what matches.

Conv#conv[kernel_shape=3,3] -> BatchNormalization#bn -> Relu#relu
(Conv|MatMul)#compute        # either op matches
*#any -> Relu#act            # wildcard for a single node

Four commands change a matched site. ! deletes a node and rewires its consumers to its input. @ sets an attribute. + inserts a node. > moves an edge between named ports. Insertions can declare an operator domain (domain=com.microsoft), an output count (outputs=3), and constant payloads loaded from files (file=weights.npy), which turned out to be the whole vocabulary needed for the case studies below.

When a rewrite builds more than a node or two, transform blocks are the readable form. A block declares the new nodes and the dataflow among them, then compiles down to the same core commands. replace mode retires the matched node and augment mode keeps it. Range syntax such as head[0..11] expands to twelve indexed nodes with ports assigned automatically. In this project the convention is to write blocks whenever possible and drop to bare commands only for a pure edge splice, which inserts nothing and therefore has nothing for a block to declare.

The language stops at structure, on purpose. Rules select sites and rewire edges. Weight arithmetic, training configuration, and quantization math do not appear in the grammar. A Python API hands the matched sites to your own code for that work.

A first pass over a model takes a few lines. Write rules as a string or a .rune file, parse them, and apply. The result object tells you how many sites matched and lists anything the gate refused.

import onnx
from byteinfer.rune import parse_rules, apply_rules_with_tracking

model = onnx.load("model.onnx")

rules = parse_rules("""
# remove identity nodes
Identity#id -> ! id

# cancel back-to-back transposes
Transpose#t1 -> Transpose#t2 -> ! t1 t2
""")

optimized, result = apply_rules_with_tracking(model, rules)
print(result.patterns_matched, "sites matched")
print(len(result.rejected_matches), "refused by the gate")
onnx.save(optimized, "optimized.onnx")

The same work is available from the command line, along with validation against the original model and performance profiling.

# apply a ruleset, or one of the predefined optimization profiles
rune-cli optimize model.onnx --rules my_rules.rune --output optimized.onnx
rune-cli optimize model.onnx --profile mobile_inference

# check the rewritten model computes the same numbers
rune-cli validate optimized.onnx --original model.onnx

# see where the time goes, CPU or GPU, with a Chrome trace
rune-cli profile model.onnx --backend ort-gpu --chrome-trace

Beyond this there are attribute constraints, regex name matching, capture groups that extract subgraphs with their weights, pattern aliases, and the transform blocks used throughout this paper. docs/dsl_feature_guide.md covers the features with examples, and docs/grammar.md has the formal syntax.

5How a rewrite is applied

Rules written in block syntax are compiled to core commands and parsed. The matcher walks the model and enumerates sites. On models past 500 nodes it switches to a pattern-specific traversal that keeps memory bounded; the 1,167-node BERT matches in seconds. Each matched site's commands become a mutation plan. The plan is applied to a shadow copy of the graph structure and checked against the five invariants in section 6. Plans that pass reach the editor and mutate the protobuf. Plans that fail are dropped and reported. A ruleset that half-applies tells you which half and why.

6The validity gate

The gate answers one question per site. If this plan runs, is the result still a complete graph? Five invariants define complete.

  • Every required input of every surviving node has a producer, an initializer, or a graph input behind it.
  • No surviving node reads a tensor whose producer the plan deletes.
  • Every declared graph output is still produced.
  • The rewired graph has no cycles.
  • Input and output counts stay legal for each operator's schema.

The invariants are cheap to state. The work is in checking them against what the editor will actually do, so the shadow copy reproduces the editor's exact semantics for delete-bypass, producer replacement, output stealing, and attribute-to-input conversion. Because the check mirrors the editor, a plan that clears the gate applies the way it simulated.

As a stress test, the gate was run against a large ruleset of structural rewrites across five benchmark models. It refused 165 applications, each one a rewrite that would have silently corrupted a residual connection while still producing a model that loads. Every ruleset in this paper applies with zero rejections, and the test suite enforces that.

The guarantee is structural. A gate-clean rewrite can still compute the wrong numbers, so checking outputs against the original model is its own step, done with the validation tools from section 4. Backends can extend the gate with their own constraint providers, say operator-set membership for a mobile target, without touching the engine.

7Case study: LoRA adapters on an exported LLM

You have a decoder exported to ONNX. You want LoRA adapters in that graph, trained there, and folded away before serving. The usual tooling assumes you still have the PyTorch model. If the ONNX file is what you have, every step is graph surgery, and on this model that means 60 separate injection sites, each with its own weight shapes.

One rule handles the q projections. It matches the projection spine, builds the adapter chain in a transform block (augment mode, so the original projection stays), and rewires the downstream Reshape to read the merged sum. A second rule does the same for v projections with a different tail. Applied to SmolLM-135M, the two rules matched all 60 sites with zero gate refusals.

Each step of the lifecycle checks the one before it, which is why we lead with this study.

Inject. Standard LoRA initialization sets A to random normal and B to zero, so the adapter path contributes nothing at first. The injected model should therefore match the original bit for bit, and it does. Measured maximum output difference against the base model, 0.0.

Train. The injected file went straight into ONNXRuntime's training API. Of 453 initializers, the 120 adapter tensors were marked trainable and the rest frozen. After 100 AdamW steps on a small local corpus, held-out NLL dropped from 2.764 to 2.608, perplexity from 15.86 to 13.58.

Merge. A 30-line numpy function folds each trained adapter into its base weight (W′ = W + 2.0·A·B), then a one-line splice rule reconnects the projection to its Reshape and the orphaned adapter chain is pruned. The merged model's dataflow topology is identical to the original base model, and its outputs match the trained adapter model within 5×10⁻⁴, the noise floor of reordered fp32 accumulation.

The v projections need their own rule. They are grouped-query projections with output width 192 rather than 576, so they take a different B matrix. Point a rule at the wrong shape and the mismatch surfaces immediately as a dimension error at the merge, not as a silent numerical bug downstream.

# q-projection rule (v rule differs in spine and B file)
MatMul#p -> Reshape#r -> Transpose#t -> Mul#rc -> Add#ra -> MatMul#qk

transform p augment {
    nodes:
        Constant#la file=rulesets/cache/lora_A_r8.npy
        Constant#lb file=rulesets/cache/lora_B_r8.npy
        Constant#lscale value=2.0
        MatMul#ma
        MatMul#mb
        Mul#ls
        Add#lmerge
    flow:
        p.input[0] -> ma.input[0]
        la.output[0] -> ma.input[1]
        ma.output[0] -> mb.input[0]
        lb.output[0] -> mb.input[1]
        mb.output[0] -> ls.input[0]
        lscale.output[0] -> ls.input[1]
        p.output[0] -> lmerge.input[0]
        ls.output[0] -> lmerge.input[1]
        lmerge.output[0] -> r.input[0]
}

# merge splice, run after the numpy fold
MatMul#ma -> MatMul#mb -> Mul#ls -> Add#lmerge -> Reshape#r
> lmerge.input[0] -> r.input[0]

The rules say nothing about rank or learning rate, and the fold arithmetic lives in the driver script. The consecutive MatMul pair at the head of the merge pattern only ever occurs in adapter chains, so the splice is safe to run globally.

Together with a companion rule that turns the 12 tanh-GELU sites into FastGelu, this makes BERT-Squad 5.07% faster on CPU than ONNXRuntime's own maximum optimization level, with outputs matching to 1×10⁻⁵. Five measurement sessions put the margin between +3.4% and +5.4%. Profiling the fused model shows MatMul at 90.8% of remaining time, so there was not much else on the table.

9Case study: attention block to GroupQueryAttention

Each attention layer in the SmolLM export is 126 dataflow ops covering rotary embedding, grouped key/value expansion, scores, masking, softmax, and the context product. The target is one GroupQueryAttention node from ONNXRuntime's contrib set. A linear pattern cannot name the q, k, and v branches in a single match, so the rewrite runs in stages. Three splice rules first give the tail direct edges to the projections. The intermediate graph is structurally valid at each stage even though nothing ever executes it, and because the gate holds throughout, a mistake in stage 2 shows up as a refusal in stage 2 rather than a corrupted model after stage 4. The final rule then matches the spliced tail and swaps in the fused node.

# final stage; stages 1-3 are one-line splices
MatMul#qk -> Mul#scale -> Add#maskadd -> Softmax#sm
  -> Cast#c1 -> Cast#c2 -> MatMul#av
  -> Transpose#ot -> Reshape#osh
+ GroupQueryAttention#gqa domain=com.microsoft
    outputs=3 num_heads=9 kv_num_heads=3 do_rotary=1
+ Constant#cosc file=rulesets/cache/smollm_gqa_cos_half.npy
+ Constant#sinc file=rulesets/cache/smollm_gqa_sin_half.npy
+ Constant#slk  file=rulesets/cache/smollm_gqa_seqlens.npy
+ Constant#tsl  file=rulesets/cache/smollm_gqa_totalseq.npy
> qk.input[0] -> gqa.input[0]   # q_proj, via splice
> qk.input[1] -> gqa.input[1]   # k_proj
> av.input[1] -> gqa.input[2]   # v_proj
> slk.output[0] -> gqa.input[5]
> tsl.output[0] -> gqa.input[6]
> cosc.output[0] -> gqa.input[7]
> sinc.output[0] -> gqa.input[8]
> gqa.output[0] -> osh.output[0]

In fp16 on an RTX 4090 the fused model runs the 256-token forward pass in 5.62 ms against the stock conversion's 6.23 ms, a 9.78% improvement, and it tracks the fp32 original as closely as the stock conversion does (mean error 0.0126 vs 0.0125, next-token agreement 99.2% for both). This target is fp16-specific: GroupQueryAttention has no fp32 CUDA kernel.

10Case study: a serving pair from one export

The GQA-fused model in section 9 is still a batch scorer: generating a token means re-running all 256 positions. Real deployments serve decoders as two fixed-shape graphs, a prefill that processes the prompt once and a decode step that processes one token against a KV cache. This pipeline derives both from the single export, with rules end to end: the fusion ruleset, then one ruleset per variant. The fused GQA nodes already carry unused cache ports, so prefill is a pure promotion (expose the 60 present tensors that were being thrown away) and decode wires new cache inputs into the empty ports plus one shared seqlens_k input. All three stages apply with zero gate refusals (120, 30, and 30 matches), and the prefill graph's logits are bit-identical to the fused base.

# decode ruleset (prefill differs only in having no nodes: section)
GroupQueryAttention#gqa

transform gqa augment {
    nodes:
        input#pk dtype=float16 shape=[1,3,256,64]
        input#pv dtype=float16 shape=[1,3,256,64]
        input#sl name=seqlens_k dtype=int32 shape=[1]
    flow:
        pk -> gqa.input[3]
        pv -> gqa.input[4]
        sl -> gqa.input[5]
        gqa.output[1] -> graph.output dtype=float16 shape=[1,3,256,64]
        gqa.output[2] -> graph.output dtype=float16 shape=[1,3,256,64]
    checks:
        validate_cache_io(gqa)
}

The anonymous pk/pv inputs get a unique name per layer; the named seqlens_k is created once and shared by all 30. validate_cache_io is a registered Python function that checks the declared cache geometry against the model's real rotary caches before any site is touched. The total_sequence_length constants stay baked at capacity; only the sequence position becomes a runtime input.

Correctness first: over 4 prompts and 128 generated tokens each, the prefill-then-decode chain produced the same 512 greedy tokens as re-running the full window every step, and both track the fp32 reference equally (mean peak logit delta 0.037 vs 0.038). What it buys, measured on the RTX 4090 with a 64-token prompt:

Two sessions do not mean two copies of the weights. Because both variants keep identical initializer names (they come from one fused parent, a property the pipeline preserves by construction), the serving driver loads the weights onto the GPU once and hands both sessions the same buffers. Measured process VRAM lands at 906 MB, at parity with the single static session. Without sharing the naive pair costs 1,460 MB; with plain run() feeds instead of IOBinding the decode step is 3.31 ms/token, still 1.79×. What remains on the cost side is the doubled disk footprint, a slower one-time init, and the prefill cache readout in first-token latency.

12Agentic evaluation

The four operator runbooks that ship with Rune — set up, stage, author, execute — are written for a person, but they also serve as the interface an autonomous agent uses. This matters because a language model can already write imperative graph-surgery scripts today; what stops that from being deployable is the failure mode. An imperative script fails silently and non-locally, and an agent's mistake is confident wrongness. The validity gate changes the shape of that failure: an agent's bad rewrite comes back as a refusal naming the rule, node, port, and invariant, not as a corrupted graph that breaks three transforms later. A machine-readable refusal is to a rule-authoring agent what a compiler error is to a code-authoring agent — the signal that makes the loop converge. The gate is the contract with the machine; the audit report is the contract with the human who reviews and approves what the machine produced.

To test this rather than assert it, two transformations were driven entirely by separate agents working from the runbooks alone, with no access to the authoring conversation and no attack vectors supplied. Each was scored by a mechanical evaluator, not by the agent's own report.

Trial 1 — Conv+BatchNorm fold, a model the project had never touched

Given only the runbooks and a ShuffleNet-v2 export, a fresh agent ran the staging analysis, found that all 56 convolutions were each followed by a BatchNormalization, computed the exact affine fold in a numpy driver, and authored the fusion ruleset. It applied gate-clean on the first attempt — 56 of 56 sites, zero refusals — taking the graph from 261 to 205 nodes with bit-identical outputs, and wrote its own regression tests. It resolved the one genuine design question (whether it could add a bias input to a bias-less convolution) by reading the operator schema rather than guessing, because the runbooks told it the gate would judge the result.

Trial 2 — INT8 quantization prep for an NPU-style target

The second task was harder in the ways that matter for real deployment: insert QuantizeLinear/DequantizeLinear pairs around every convolution under a first-and-last-layer exclusion policy, with quantization scales derived from a real calibration pass, on an opset-10 model that permits only per-tensor scales. The agent discovered the opset constraint itself, wired the exclusion with a negative name constraint, and hit the gate's check hooks fifty-five times in one wave when its own numeric guard caught an attempted re-quantization of already-quantized sites — a refusal it read and corrected by bounding the pass, never a corrupted output. No structural invariant was ever breached. When plain min/max weight scaling scored zero of eight on top-1 agreement, the agent independently diagnosed that BatchNorm folding had created outlier weight channels, then applied percentile clipping and post-training bias correction to reach the acceptance bar. The result passes a node-by-node audit of QDQ convention and is recognized by ONNX Runtime's own quantization machinery, which fuses it into 35 integer convolution kernels.

Both trials also produced something the authors did not: precise reports of where the runbooks were wrong or incomplete in a fresh context. Those gaps were fixed. The point of the exercise is narrow and load-bearing — the same structural guarantee that makes a human's rewrites safe is what makes an agent's rewrites safe enough to run unattended, and the reports it emits are what let a human sign off on the result.

13Adversarial testing

The structural safety contract is the product, so it is attacked deliberately rather than only exercised. Every attack is judged against a single invariant: an operation must end in exactly one of two states — clean success with a structurally valid output model, or a clean refusal or error with the input graph left untouched. A crash, a hang, a half-applied mutation, a refusal that nonetheless changed the graph, or an invalid model presented as success are all defects, no matter how hostile the input was.

The attacks come from two sources that find different things. The first is a fixed suite spanning the entire language surface — the parser, the block compiler, the gate itself, atomicity under partial failure, the check hooks, per-match templates, multi-input and fan-in/fan-out topologies, wildcards, ranges, opset-dependent insertion — plus a seeded random fuzzer that generates graphs and rules and checks that every output either loads or was cleanly refused. This suite tests the failure modes the authors could imagine.

The second is an unanchored adversary: a separate agent given the invariant and told where the fixed suite already looked, but handed no attack vectors of its own. Its job was the failure modes the authors could not imagine — feature interactions, state that persists across matches, and above all the seams where the gate's safety proof (computed on a shadow of the graph) and the editor that performs the real mutation might disagree. It was required to be its own skeptic, refuting each candidate against the documented behavior before reporting it.

Together the two rounds have run roughly ninety distinct attacks and found ten genuine engine defects, each since fixed and pinned as a permanent regression. The most significant was found by the unanchored adversary and not the fixed suite: a single construction — inserting an operator that requires a variable number of inputs but wiring none of them — that the gate would have admitted, shipping a model no runtime could load. It was reproduced, traced to a gap in how the gate modeled minimum operator arity, fixed, and turned into a regression test the same day. That is the intended shape of the process. The claim Rune makes is not that the gate is perfect; it is that violations of the safety contract are found and closed, and that the search for them is systematic on both the axes a human can enumerate and the ones only an adversary discovers.

14Current scope and roadmap

The engine today covers the workflows shown in this paper: linear patterns with wildcards and constraints, transform blocks, staged multi-branch rewrites under the gate, and the validation and profiling CLI. When you plan a project around it, a few things are useful to know.

  • Rewrites that bind several branches at once, like the attention fusion, are staged through splice rules today, with the gate keeping every intermediate state valid. Direct multi-branch patterns are on the roadmap.
  • Multi-node wildcards are bounded to five nodes, which keeps matching fast on large models.
  • The gate guarantees structure. Numerical equivalence is a separate check, run with the bundled validation tools.
  • The numbers in section 11 were measured on x86-64 and an RTX 4090. Margins vary with hardware, so confirm them on your target with the built-in profiler before shipping.
  • The project is in active development. 496 automated tests stand behind the engine and every example in this paper, and interfaces may still evolve between releases.

15Where it fits

The runtime's built-in optimizer should always run first; it fuses a fixed catalogue of patterns for free. ByteInfer Rune takes the patterns that catalogue misses. The TensorFlow LayerNorm and the attention fusion above are both cases ONNXRuntime leaves untouched at maximum optimization.

onnx-graphsurgeon and hand-rolled protobuf scripts give the same reach with no safety net, and validity only shows up at session build. ByteInfer Rune gives up none of that reach and adds the gate, which is what keeps a 60- or 120-site rewrite reviewable instead of a leap of faith.

MLIR and TVM Relay offer stronger pattern formalisms, but they live inside full compiler stacks with their own IR. ByteInfer Rune works on the ONNX file you already ship, with no migration.

Superoptimizers such as TASO and PET search for which rewrite to make. ByteInfer Rune applies a rewrite you already want, safely, at every site of a real model.
